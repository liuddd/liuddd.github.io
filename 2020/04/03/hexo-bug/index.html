hexo d命令推送远程失败

今天在本地新增hexo文章后，使用hexo d推送至远程，经历了一系列报错，最终成功解决。记录一下日志。

Round 1 —— 推送远程无反应：配置deploy选项

hexo clean、hexo g、hexo d推送至远程，先是没有任何反应，即没有返回信息也不报错，这种情况可能是项目配置文件有问题，cat _config.yml发现deploy选项未进行配置，如下

于是执行vim _config.yml进行配置，如下

type设置为git，repo设置为项目的git仓库地址，branch设置为项目所在的分支

Round 2 —— Deployer not found：安装hexo-deployer-git插件

经过这一步，我再次尝试hexo d，报错Error Deployer not found: git，如下

这是因为没有安装hexo-deployer-git插件，安装就行，执行命令npm install hexo-deployer-git --save

Round 3 —— Could not read form remote repository：添加github.com的本地DNS解析

经过这一步，再次尝试hexo d，依然报错，错误信息

git@github.com: Permission denied (publickey).

fatal: Could not read from remote repository.

可是经检查我的本地生成SSH密钥已经配置到github账号中去了 ，那么为什么无法读取远程仓库呢？

执行命令ssh -T git@github.com发现连接超时，执行ping github.com发现无法ping通，也就是说本地DNS无法解析

在网络中ping是一个十分强大的TCP/IP工具。它的作用主要为：

（1）用来检测网络的连通情况和分析网络速度；

（2）根据域名得到服务器IP；

（3）根据ping返回的TTL值来判断对方所使用的操作系统及数据包经过路由器数量。

那么我们去到hosts文件里看看是不是没有把github.com的域名放进解析列表里，执行cat /etc/hosts浏览文件内容，发现的确没有github.com的解析，执行vim /etc/hosts在文件中添加如下内容

注意IP地址和域名中间有且只有1个空格，我之前就多打了空格导致无法解析。这样的格式错误很容易被忽略，却很难发现问题所在。

加上之后，执行ping github.com发现可以ping通了，本机可以正常解析github.com这个域名了。

Round 4 ——git Permission denied(publickey)：添加自定义密钥名到ssh

再次尝试ssh -T git@github.com，仍然报错git@github.com: Permission denied (publickey).查阅资料后得知，可能是密钥命名的问题，一般我们有多个git账号时，需要在本机配置多个SSH-key，而我的习惯是通过不同命名来标记不同账号，比如我会通过ssh-keygen -t rsa -C ‘13077370662@163.com' -f ~/.ssh/github_id_rsa命令生成我的github账号专属的密钥，而github_id_rsa这个名字又和ssh内设定的名字不一致，所以ssh会拒绝权限，解决方法是执行ssh-add ~/.ssh/你的名字，将我起的这个名字加入ssh中，这时候再执行ssh -T git@github.com就可以看到成功返回信息了

最后执行hexo d发现已经成功将最新渲染的页面push到远程，github上的项目已经同步更新。到这里，问题终于解决了。
